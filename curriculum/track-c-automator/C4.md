# Make it run without you: triggers, review gates, and graceful failure

## 1. Hook (≈15s read)
You've mapped it (C1), delegated it (C2), and split it into a team (C3). The last step turns it into **{{deliverable}}** for real: making it run **on its own** — on a schedule or a trigger — while still protecting you when something goes weird in {{industry}}. That's a true automation, not a faster way to do it by hand.

## 2. Concept
A hands-off automation has three parts beyond the task itself. The framework is the **operating model**:

**Framework — TRIGGER → GATES → ERROR-HANDLING → LOG (the hands-off operating model):**

1. **TRIGGER** — what makes it run without you: a **schedule** ("every Monday 7am") or an **event** ("a new return arrives," usually via a connector/MCP).
2. **GATES** — the human review checkpoints kept on anything irreversible, so "unattended" never means "unsupervised on the dangerous parts." (These are your C2 gates, now running on a schedule.)
3. **ERROR-HANDLING** — what happens when an input is malformed or a step fails: it should **skip + flag**, **pause + notify**, or fall back to a safe default — **never fail silently** and never barrel ahead.
4. **LOG** — a short run summary (processed · escalated · skipped · links to items needing you) so you can glance and *trust* it's working. No observability = you find out it broke from an angry customer.

The mindset shift is from *running* the workflow to *operating a system that runs it.* *(Scheduling and event triggers depend on your tools/connectors and evolve — check current Claude docs for what's available; the operating model above is the durable part.)* Launch **notify-only or draft-only** first; promote to fuller autonomy only after a clean week of logs.

## 4. Do it (the core — hands-on)
Take your C2/C3 automation and design its hands-off version. (You're writing the *operating plan*; wiring the live trigger depends on your tools — design it fully and connect what you can.)

```
Turn my automation into a hands-off system. Here's the task: [PASTE your C2/C3 brief].
Design and document, using the operating model:
1. TRIGGER: when/how it runs without me (schedule or event). Suggest a sensible cadence.
2. GATES: which steps must pause for my approval (irreversible / customer-facing / over a threshold).
3. ERROR-HANDLING: for each likely failure (bad input, missing data, a step fails) — skip+flag /
   pause+notify / safe default. Never fail silently.
4. LOG: the short run summary I should get each time (what to include so I can trust it at a glance).
Give it back as a one-page "automation operating plan" I can implement.
```

Then run the task once *as if triggered* and produce a sample run log. Confirm the gates and the "couldn't process" handling behave the way your plan says.

> **Privacy & safety note:** unattended automations touching money, customers, or records need the *tightest* gates and clearest logs. Never schedule an irreversible action without a gate or a hard threshold. Start notify-only/draft-only and promote once the logs look clean for a week.

- **ERROR-HANDLING** covers the likely failures with **no silent failures**.
- You've specified the **LOG** and validated it with a sample run.

### ⚠️ Common mistakes
- **No gate on irreversible actions.** A scheduled refund/send with no gate is how automations cause real damage.
- **Silent failure.** If a bad input just disappears, you'll trust a broken system. Flag everything it can't handle.
- **No run log.** "Set and forget" without observability means you find out it broke from a customer.
- **Going fully autonomous on day one.** Run notify-only/draft-only first; widen autonomy as the logs prove it.

### 🔎 Honest limit
Unattended is the highest-risk mode: a mistake now happens *without you watching*, repeatedly, on a schedule. AI steps can still misjudge an edge case, and connectors/triggers can fail. That's why the gates, error handling, and logs aren't extras — they're what makes hands-off responsible. Keep money-moving and customer-facing actions gated, monitor the logs, and treat the automation as a system you *own and audit*, not one you forget.

### 📊 Case study — "autonomy is earned" is an industry rule, not a slogan
**Grounded in agent-safety practice.** Anthropic's *Building effective agents* makes the point explicitly: as you give an agent more autonomy, you trade away predictability, so production systems keep humans in the loop at the consequential moments and add guardrails around what the agent can do unattended. This mirrors a hard-won rule from site-reliability and operations engineering: a system you can't *observe* is a system you can't trust — which is exactly why LOG is a first-class part of the model, not an afterthought. The failure mode is always the same shape: a scheduled action with no gate and no log runs wrong for a week before anyone notices, usually via an angry customer. TRIGGER → GATES → ERROR-HANDLING → LOG is the seatbelt that makes "runs without me" responsible rather than reckless.
