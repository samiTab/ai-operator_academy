# Stop re-typing: build your prompt library — and your first Skill

## 1. Hook (≈15s read)
By now you've written prompts that work — for your inbox, your documents, your research in {{industry}}. The amateur retypes them from memory every time. The operator keeps a **library** and lets the best ones run *automatically*. Today you build both — and in doing so, you lock in the core of **{{deliverable}}** as a reusable toolkit, not a one-session memory.

## 2. Concept
Two levels of reuse, smallest first:

**A prompt library** is just your proven prompts, saved with `[BRACKETS]` for the parts that change, organized so you can grab one in seconds. That alone compounds every hour you've invested in this track.

**A Skill** is the next level: a small package of your know-how — instructions plus optional templates/examples — that Claude **applies automatically when a task matches**, so you don't even paste a prompt. In Claude apps and Claude Code, a Skill is a folder with a `SKILL.md` file: a name, a short description of *when to use it*, and the instructions Claude should follow. *(This very academy is built from six Skills.) The mental model: a saved prompt is a note to yourself; a Skill is a trained specialist Claude becomes on demand. *(UI for creating Skills evolves — check the latest Claude docs for the exact steps; the structure below is the durable part.)*

**Framework — COLLECT → TEMPLATE → PACKAGE → TRIGGER (the reuse ratchet):**

1. **COLLECT:** Every time a prompt works, save it immediately under a clear header. Don't curate yet — capture first. A prompt you can't find tomorrow is a prompt you'll retype.
2. **TEMPLATE:** Replace all specific values with `[BRACKETS]`. The client name, the project, the topic — anything that changes per use gets a bracket. This is the move that turns a one-off prompt into a tool.
3. **PACKAGE:** Promote your most-used templated prompts into Skills — write the `SKILL.md` with a name, a precise *when-to-use* trigger, and the ordered instructions. One Skill per job.
4. **TRIGGER:** Add the Skill, then test it by triggering it *naturally* (say what you want, not a command). Confirm it fires on the right tasks and produces your standard output without pasting.

## 4. Do it (the core — hands-on)
**Part 1 — library (everyone).** Open a doc called *My Prompt Library*. Paste the 3–5 prompts you've saved across F2–A3, each under a clear header, each with `[BRACKETS]` for the variable parts. That's your library — done.

**Part 2 — your first Skill (promote your best one).** Take the single prompt you'll use most and write it as a Skill:

```
---
name: my-[task]-skill           e.g. proposal-drafter
description: Use this when [the exact trigger — e.g. "I ask to draft a client proposal"].
             [One line on what it produces.]
---

# [Skill name]

When this applies, do the following:
1. [step — what context to use / ask for]
2. [step — the structure/format of the output]
3. [step — the style/voice and any must-includes]

Always: [your non-negotiables — e.g. "flag any number you're unsure about"].
```

Add it as a Skill in your Claude app (see the current docs for the exact "create Skill" step), then trigger it naturally — *"draft a proposal for [client]: [notes]"* — and confirm Claude applies it without you pasting the prompt.

> **Privacy note:** keep credentials and confidential client data *out* of saved prompts and Skills — use `[BRACKETS]` placeholders and paste the sensitive specifics only when you run it.

### 🎁 Keepable asset — your library starter + Skill template
```
=== MY PROMPT LIBRARY ===
## [Task name]   — use when: [trigger]
[the prompt, with [BRACKETS] for variable parts]
(repeat per task)

=== SKILL TEMPLATE (SKILL.md) ===
---
name: my-[task]-skill
description: Use this when [trigger]. [What it produces.]
---
# [Skill name]
1. [context to gather]   2. [output structure]   3. [style + must-includes]
Always: [non-negotiables].
```

### ⚠️ Common mistakes
- **A vague Skill description.** Claude decides *when* to apply a Skill from its description — if it's fuzzy, the Skill won't fire (or fires at the wrong time). Name the trigger precisely.
- **Baking in specifics that change.** Use `[BRACKETS]`; a Skill hard-coded to one client is useless for the next.
- **Cramming five jobs into one Skill.** One Skill, one job — small and sharp beats big and vague.
- **Library sprawl with no headers.** If you can't find a prompt in 5 seconds, you'll just retype it.

### 🔎 Honest limit
A Skill is only as good as the instructions you put in it — Claude follows your steps, so a flawed or ambiguous Skill produces flawed output *consistently*. Skills also don't make Claude smarter about your private facts; they package *how* you want work done, not *what's true today*. Review a new Skill's output a few times before you trust it on autopilot, and keep the privacy placeholders honest.
