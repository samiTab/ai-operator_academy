# Connect your tools (MCP): give Claude hands into the apps you already use

## 1. Hook (≈15s read)
So far Claude works with what you paste in. The last Builder skill removes the copy-paste: **connect** it to the apps you already run in {{industry}} — calendar, docs, your invoicing tool — so it can read and act there directly. This is the difference between an assistant that drafts and one that *does* — and the connector pattern is how **{{deliverable}}** moves from a tool that helps you to a system that runs.

## 2. Concept
**MCP (Model Context Protocol)** is an open standard for plugging Claude into external tools and data sources — calendars, file stores, databases, SaaS apps. In Claude's apps these often show up as **connectors** you enable; under the hood, an MCP "server" exposes a tool (say, your invoicing system) and Claude can call it. The point: Claude stops being a sealed box and gets *hands* into your real systems. *(Which connectors exist, and how you add an MCP server, change frequently — follow the current Claude "connectors / MCP" docs. The durable ideas are below.)*

**Framework — GOAL → SCOPE → READ → GATE (the safe connector checklist):**

1. **GOAL:** Name exactly what you want Claude to reach in which app, and *why paste won't do*. If copy-paste works, you don't need a connector. Connections add risk; use them only when the live-data or action value is real.
2. **SCOPE:** Choose the *minimum access* that achieves the goal — read-only before read-write, one app before many. Never grant full access "to be safe" — that's the opposite of safe.
3. **READ:** Connect and confirm reads work correctly before enabling any write or action. "It reads my calendar correctly" is a milestone. Only cross to actions after that milestone is solid.
4. **GATE:** Define explicitly which actions require your manual approval before they execute — anything send, pay, delete, or customer-facing stays gated until you've seen it work many times.

Trust the source is the baseline: only add MCP servers/connectors you trust — they can see whatever you grant.

## 4. Do it (the core — hands-on)
Pick one workflow in {{industry}} that's stuck because Claude can't reach a live app. Connect **one** tool, safely:

```
GOAL: let Claude [read / draft-act in] [ONE app] for [this workflow].
WHY NOT PASTE: [what's live or changing that makes paste impractical]
PLAN with me first:
1. Which connector/MCP server do I need, and is it one I trust?
2. What's the MINIMUM access? (start READ-ONLY if possible)
3. What will I ask it to do once connected — and which actions stay behind my approval?
Then walk me through connecting it (current docs), and we'll test with a READ-ONLY question first.
Only after that works do we try a draft/action — never an irreversible one unattended.
```

If you can't connect a tool right now (some need admin/setup), still produce the **plan + permission decisions** and test the read-only step where possible.

> **Privacy & safety note:** a connector can expose real business data and take real actions — grant least privilege, prefer read-only while learning, only connect sources you trust, and keep send/pay/delete behind a manual approval. Review what each connector can access before enabling it.

### 🎁 Keepable asset — the "Safe Connector" checklist
```
Before connecting a tool to Claude (GOAL → SCOPE → READ → GATE):
□ GOAL:  Do I actually need live data/actions, or would paste work? (if paste works, skip)
□ SCOPE: MINIMUM access for the job — read-only first; never full access "to be safe"
□ TRUST: Is this connector/MCP server from a source I'd trust with my data?
□ READ:  Confirm reads are correct BEFORE any write or action
□ GATE:  send · pay · delete · customer-facing → always behind MY manual approval
□ Review what it can access; revoke if unused
```

### ⚠️ Common mistakes
- **Granting full access "to be safe."** That's the opposite of safe. Least privilege, always.
- **Connecting untrusted servers.** A connector sees what you grant — vet the source.
- **Letting it act before you've tested reads.** Confirm it reads correctly before it does anything.
- **Using a connector when paste would do.** Connections add risk; use them only when the live-data/action value is real.

### 🔎 Honest limit
Connectors widen what can go wrong: a misjudged action now happens in a *real system*, and a connector with broad access is a real security surface. MCP also doesn't make Claude infallible — it can call a tool with a wrong assumption. Keep scopes minimal, irreversible actions gated, and connectors limited to trusted sources; for anything touching customer data, payments, or production systems, apply the governance practices in Track D before you widen access.

### 📊 Case study — an open standard, not a lock-in
**A real industry shift.** Anthropic introduced and open-sourced the **Model Context Protocol** in November 2024 as a universal way to connect AI assistants to the tools and data where work actually lives — described at the time as "a USB-C port for AI applications." It wasn't kept proprietary: through 2025 MCP was adopted across the industry by other major AI providers and a fast-growing ecosystem of connectors. For an operator that matters two ways: the connector skill you learn here is **portable** (not bet on one vendor), and the *reason* the GOAL→SCOPE→READ→GATE checklist exists is that an open, powerful connection standard puts real systems one instruction away — so least-privilege and gates aren't bureaucracy, they're the seatbelt.
