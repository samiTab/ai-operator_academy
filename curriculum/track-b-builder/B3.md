# Agent Skills: package your know-how so Claude just *does it your way*

## 1. Hook (≈15s read)
You have a "right way" to do certain things in {{industry}} — your quote format, your brand voice, your checklist. Re-explaining it every time is the tax. A Skill pays it off once: you package the know-how, and Claude applies it automatically whenever the task comes up — for you, for your team, and as a pillar of **{{deliverable}}**.

## 2. Concept
A **Skill** is a folder Claude can use — at its heart a `SKILL.md` file with a **name**, a **description of *when* to use it**, and the **instructions** to follow; optionally bundled with **templates, examples, or reference files** the Skill pulls in. When a task matches the description, Claude loads the Skill and follows it — so your standards travel with the work instead of living in your head. *(This academy itself is built from six Skills. The exact "add a Skill" surface differs across Claude apps and Claude Code and evolves — check current docs; the `SKILL.md` structure is the durable core.)*

**Framework — TRIGGER → BUNDLE → TEST → REFINE (the reliable Skill loop):**

1. **TRIGGER:** Write the when-to-use description before anything else. Be precise: *"Use this when I ask to write a client quote"* beats *"Use this for writing."* The trigger is how Claude decides when to load the Skill — fuzzy trigger = wrong-time fires or misses.
2. **BUNDLE:** Attach a real template, approved example, or reference file that shows what "correct" output looks like. *Showing* is more reliable than *describing* — Claude matches format to an example with far less drift.
3. **TEST:** Run the Skill on two real cases — one straightforward, one edge case (a discount, an exception, an unusual request). Don't ship untested; a Skill applies *consistently*, so an untested flaw repeats every time.
4. **REFINE:** Edit the `SKILL.md` instructions based on what the tests reveal. One round of refinement is usually enough to close the gap. Then it's production-ready.

If you took A4, you wrote a basic single-prompt Skill. B3 goes a level deeper: bundle assets, write a precise trigger, and refine from real tests.

**Second example — consulting firm, client-deliverable template.** Priya is a partner at a boutique consulting firm where every associate writes the executive summary section differently, and it costs her an hour of editing per deck. She builds an `exec-summary/SKILL.md`: TRIGGER ("use when I ask to write an executive summary for a consulting deliverable"), BUNDLE (attaches an approved 1-page exec summary from a past engagement as the reference format), TEST (runs it on two real briefs — the first is fine, the second is a transformation project with a quantified ROI that needs a different structure in the headline section), REFINE (adds one instruction line: "if the deliverable includes a quantified ROI, lead the summary with it"). Both tests pass. She rolls it to the team. **~1 hr of editing per deck → ~10 min spot-check.** The TRIGGER→BUNDLE→TEST→REFINE loop travels to any industry where you have a "right way."

## 4. Do it (the core — hands-on)
Pick a repeated, *opinionated* task in {{industry}} (one where you have a "right way"). Build a real Skill:

```
Help me build a Skill for [TASK].
1. TRIGGER — "Use this when [precise situation]": write it so it fires for the right requests
   and NOT for unrelated ones.
2. BUNDLE: I'll attach [my template / an approved example / reference doc] — write instructions
   that tell Claude to match its format.
3. INSTRUCTIONS: the exact steps / rules / my "right way" for this task.
Then: write the SKILL.md, I'll add it, and we TEST it on 2 real cases — including one tricky
edge case — and REFINE the instructions until both come out right.
```

The deliverable is a Skill that **passes two real tests**, including an edge case you had to refine for.

> **Privacy & safety note:** keep secrets and confidential client data *out* of Skill files (they get reused and may be shared) — use placeholders; bundle only assets you're comfortable reusing.

## 8. ROI beat
A Skill removes the re-explain tax *and* enforces consistency: every output meets your standard without you checking. On a task done ~10×/week, that's ~**1–2 hours back weekly** plus fewer corrections — and if a teammate uses it, the savings multiply across the team.

### 🎁 Keepable asset — the deeper Skill scaffold
```
folder: my-skill/
  SKILL.md:
    ---
    name: my-skill
    description: Use this when [precise trigger]. [What it produces.]
    ---
    # My Skill
    1. [context/inputs to gather]
    2. [exact steps / rules]
    3. [output format — match the bundled example]
    Always: [non-negotiables]; Never: [out-of-scope, so it doesn't over-fire].
  example.(pdf|md|xlsx)   ← the approved sample it should match
  template.(…)            ← optional starting file
TRIGGER → BUNDLE → TEST on 2 real cases incl. 1 edge case → REFINE → done.
```

### ⚠️ Common mistakes
- **Fuzzy trigger.** Too vague and it fires on the wrong tasks; too narrow and it never fires. Be precise.
- **No bundled example.** Telling beats describing — attach the real sample so format matches.
- **Shipping untested.** A Skill applies *consistently*, so an untested flaw repeats every time. Test the edge case.
- **One Skill, many jobs.** Keep it to a single job; spin up another Skill for the next.

### 🔎 Honest limit
A Skill makes Claude do things *your way consistently* — it does **not** make Claude more accurate about facts or immune to mistakes. A flawed instruction produces a flawed result every single time, at scale, which is worse than a one-off error. Treat a new Skill like a new hire's first week: supervise its output on real cases before you trust it on autopilot, and keep sensitive data out of the files.

### 📊 Case study — why "showing" beats "describing"
**Grounded in how Skills are designed.** Anthropic built Agent Skills around a principle called *progressive disclosure*: a Skill's short description is always visible to Claude, but its full instructions and bundled files load only when the task actually matches — so you can stack many specialized Skills without overwhelming the model. That design is exactly why this lesson's BUNDLE step works: attaching an approved example file isn't decoration, it's the highest-signal instruction you can give, because Claude matches a concrete sample with far less drift than it matches a written description. The same Skill folder is portable across Claude apps and Claude Code — write it once, your standard travels with the work.
