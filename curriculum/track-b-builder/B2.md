# Build a small internal tool that earns its keep

## 1. Hook (≈15s read)
A "quote generator," a "data cleaner," a "mini dashboard" — the little internal tools you'd love but assumed you'd have to buy or hire for in {{industry}}. Today you build one real one with Claude Code, scoped tightly enough to finish in a sitting — and that finished tool is a concrete piece of **{{deliverable}}**.

## 2. Concept
The difference between a frustrating build and a satisfying one is **scope**, not skill. A good first tool does **one job, with a clear input and a clear output**, for one person (you). You define those three things; Claude Code does the rest through the B1 loop — but now with two upgrades: you build **iteratively** (get the simplest version working, *then* add features) and you **describe behavior with examples** ("when I paste these line items, it should output a PDF that looks like this").

**Framework — SPEC → V1 → ITERATE → SHIP (the working tool loop):**

1. **SPEC:** Before any code, write down the tool's job (one sentence), its input (exact fields), and its output (exact shape — include a real example of what "good" looks like). A spec is buildable; a wish is not.
2. **V1:** Ask Claude Code to build the *simplest possible working version* first — no extra features. Test it immediately on a real input. "Working" means you got the output you specified, not that it's pretty.
3. **ITERATE:** Add one feature at a time. Describe it, build it, test it on real input, commit. Don't add the next thing until the current thing works. Scope creep is how first builds never finish.
4. **SHIP:** When the tool is genuinely useful for real work — not perfect, not feature-complete — use it. A tool you use is a tool that earns its keep. Perfecting it later is easier once you know what actually matters.

The mental model: a **spec, not a wish**. "Make me a dashboard" is a wish. "A page where I paste this week's sales numbers and it shows me total, top product, and a simple bar chart" is a spec — and a spec is something Claude Code can build and you can verify.

**Second example — real estate, comparable-sales tool.** Lucia is a real estate agent who spends an hour per listing manually pulling comparable sales from her MLS export and formatting a comparison table for clients. ## 4. Do it (the core — hands-on)
Use the tool idea from B1 (or pick the one most useful for {{industry}}). Write its spec, then build iteratively:

```
SPEC for my tool:
- JOB: [the one thing it does]
- INPUT: [what I give it — exact fields/data]
- OUTPUT: [what it produces — exact shape; attach an example I like]
- FOR: just me, internal use.

Plan: start with the SIMPLEST working version (input → output), nothing fancy.
Build that, run it so I can test. Then I'll add features one at a time.
```

Get v1 working and *use it on one real input*. Then add **one** improvement and re-test. Commit after each working step. Stop when it's genuinely useful — not when it's perfect.

> **Privacy & safety note:** keep it local and internal while learning; don't wire it to live customer data or the public internet yet. Commit each working version so you can always roll back.

## 5. Check
**Done when:**
- A written **spec** (job + input + output + example).
- A **working v1** you tested on a **real input**.
- **At least one iteration** added and re-tested, with **commits** along the way.

```
JOB:    the one thing it does
INPUT:  exact fields/data I provide
OUTPUT: exact shape it produces (+ an example I like)
FOR:    just me, internal
BUILD:  SPEC → V1 (simplest working) → test on real input → ITERATE one feature → re-test → COMMIT → SHIP
```

### ⚠️ Common mistakes
- **Scope creep.** "And also…" kills first builds. Ship v1, then add one thing at a time.
- **No example output.** Showing Claude Code a sample of what "good" looks like beats describing it.
- **Testing with fake data only.** Run it on one *real* input — that's where the gaps show.
- **Not committing.** Without checkpoints, a bad iteration can lose your working version.

### 🔎 Honest limit
A tool Claude Code builds in a sitting is a *useful internal helper*, not hardened software — it may break on inputs you didn't test, and it hasn't been secured or stress-tested. That's fine for personal/internal use; before anything touches customer data, live money, or the public internet, it needs real review (and the B4/Track D safety practices). Test the edge cases that matter to you, and keep it internal until it's earned more.

### 📊 Case study — "vibe coding" has a name (and a catch)
**A real, named shift.** In early 2025, AI researcher **Andrej Karpathy** (a founding member of OpenAI and former director of AI at Tesla) coined the term *"vibe coding"* for exactly what you did in B1: describing software in plain language and letting the AI write it, while you steer by results rather than reading every line. The term went mainstream within weeks. But the practitioners who use it well echo this lesson's discipline: the catch isn't writing the code, it's **scoping and testing** it. A "spec, not a wish," V1-first, and testing on real input is what separates a tool that earns its keep from a demo that breaks on the second use.
