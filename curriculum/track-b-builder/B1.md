# Claude Code without fear: you describe it, it builds it

## 1. Hook (≈15s read)
"Coding" is the word that makes most business owners click away. Forget it. Claude Code is a tool where you describe what you want *in plain English* and it writes, runs, and fixes the code for you — even in {{industry}}. Today you get set up and build one tiny real thing, and the tool you ship becomes the first concrete step toward **{{deliverable}}**.

## 2. Concept
Claude Code is Claude working **inside your computer's files and terminal**: it can read your files, write new ones, run commands, and see the results — then fix its own mistakes. You stay in plain English; it handles the syntax. Think of it as a junior developer who types fast, never gets bored, and explains everything — while *you* remain the product owner who says what to build and whether it's right.

**Framework — EXPLORE → PLAN → CODE → COMMIT (the safe builder's loop):**

1. **EXPLORE:** Point Claude Code at your goal and the folder it'll work in. Let it look around and describe its approach before touching anything. This is where it calibrates to your situation.
2. **PLAN:** Ask for an explicit plan — the files it'll create, the steps it'll take — *before* a single line of code is written. Read and correct the plan. A wrong plan costs two minutes to fix; a wrong build costs a session.
3. **CODE:** Once the plan is approved, let it build. Watch as it implements, step by step. You can intervene at any point — *"stop, that's not right, do it this way instead."*
4. **COMMIT:** After each working step, save a checkpoint (a git commit). This is your undo button. With commits, a bad iteration never loses you a working version.

The single most important habit for non-coders: **make it PLAN before it builds**, and read the plan. *(Install steps and the exact interface change over time — follow the current "get started with Claude Code" docs for your OS; the loop above is what stays true.)*

## 3. Concept check — what it's NOT
It's not magic and not autonomous-by-default: it asks permission for consequential actions, and you approve. You don't need to know what the code says — you need to know what you *want* and whether the result *works*.

## 4. Do it (the core — hands-on)
> **Setup first (sandbox=false — this runs on your machine):** follow the current official "Install Claude Code" guide for your OS, open it in a new empty folder, and confirm it responds to a hello message. (If setup blocks you, that's expected on a first run — note where you got stuck for the coach.)

Pick a *tiny* real thing — e.g. "a script that renames a folder of files," "a one-page web form that emails me the entry," "a spreadsheet cleaner for {{industry}} data." Then drive the loop in plain English:

```
1. EXPLORE/GOAL: "I want [the tiny tool], for [who/what]. Here's what 'done' looks like: [1–2 lines].
   Look around this folder first and tell me how you'd approach it."
2. PLAN: "Before writing anything, give me a short plan. What files, what steps?" → read it, correct it.
3. CODE: "Go ahead and build it. Show me as you go." → let it write and run.
4. CHECK & COMMIT: "Run it so I can see it work." Try it. Then: "Save this as a checkpoint (commit)."
```

The win today is *small and finished*: one tool that runs, built by describing it.

> **Privacy & safety note:** Claude Code acts on your real files — start in a **new empty folder**, approve actions deliberately, and don't point it at sensitive directories while you're learning. Commits are your undo button — use them often.

**Second example — healthcare admin, appointment reminder script.** A dental practice manager, Keiko, needs a script that reads a CSV of tomorrow's appointments and generates a personalized reminder message for each patient. She has never coded. She starts in a fresh folder, pastes her CSV format, and drives the loop: EXPLORE ("here's my CSV layout, here's what 'done' looks like: a .txt file with one message per patient") → PLAN (Claude proposes: read CSV, template the message, write output file — she approves) → CODE (it builds and runs; she tests on 5 real rows) → COMMIT ("save this version"). Total time: 40 minutes. She now runs the script every morning in 30 seconds. The framework is the same whether you're in trades, healthcare, or any other sector — "describe it, plan it, build it, save it" is the loop.

### ⚠️ Common mistakes
- **Letting it code before planning.** Reading a 4-line plan first saves you from a wrong build.
- **Starting in a real/important folder.** Use a fresh empty folder while learning.
- **Approving actions you didn't read.** Glance at what it's about to do; it asks for a reason.
- **Aiming too big.** First build = tiny and finishable. Ambition comes in B2.

### 🔎 Honest limit
Claude Code can write code that *looks* right but has a bug, or take a wrong approach if your goal was vague — so the plan-first habit and actually *running and testing* the result are non-negotiable. It's a powerful junior developer, not an infallible one; you own the "does it actually work?" check, and commits are your safety net to roll back.
